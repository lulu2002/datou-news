一直以來，我總會在寫程式時，冒出很多類似「我這樣寫合適嗎？」「這樣寫會不會怪怪」…等疑問， 且每次修來修去，都還是很難找到一個真正好看的撰寫方式， 直到我閱讀了一本名為《Clean Code 無瑕的程式碼》的書

還記得當時只看完第一章的內容，我就已經對作者佩服的五體投地… 沒想到連最簡單的「變數命名」也充滿這這麼多學問，緊接著還有函式的命名、Class 的分配……， 雖然後面的內容有點超出我的能力範圍，所以沒有整本看完，但光前面幾章，就大大影響了我日後程式碼編排的方式。

其中讓我印象最深刻的內容，是作者在方法編排章節所說到的：

好的程式碼，能讓讀者在瀏覽時，如同看報紙一般，由上而下自然且順暢的閱讀文章，並清楚每行程式碼所代表的意義。

一直以來，我總是習慣於將所有程式寫在同一個函式裡， 然而這種做法，只要內容一多，不但維護起來困難，會大幅增加 Debug 的時間， 且當我過個幾天再回來看當初寫的內容時，會完全搞不懂這段程式代表的意義，必須耗費很長一段時間回想， 最終造成開發效率低落。

上面那段話，給了我改進的方針，我開始將一大段程式拆分成多個內容簡短的函式， 這種寫法真的很費時，光命名就得耗費不少時間， 但隨著時間經過，我慢慢發現，當初所寫的那些程式碼，即使一兩個月沒看，也能立馬讀懂它所代表的意思， 同時因為每個函式都很簡短，用戶回報 Bug 也能馬上找到癥結所在。

在寫程式的旅途中，能遇到這本書真的是一件很幸福的事情。 雖然書中是以 Java 為範例，但很多觀念基本上適用於任何程式語言， 因為讀了這本書，我才能有機會去養成寫程式的好習慣！
